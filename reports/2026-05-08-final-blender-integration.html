01 / Executive Summary

Executive Summary

What was done

Three phases of a Blender integration pipeline were implemented for a 4×25m five-storey tube-house model generated in FreeCAD:

  1. PHASE-01: OBJ/MTL export adapter with named group conventions and 6-material MTL palette
  2. PHASE-02: Blender scene assembly with 10 PBR materials, data-driven JSON sidecar config, sun+fill lighting, camera targeting, and headless Cycles render at 1920×1080
  3. PHASE-03: Batch runner integration with separate run_blender.sh, OBJ output in the FreeCAD success summary, and end-to-end verification

Why it matters

The tubehouse design now has a complete path from parametric specification (floorplan-spec.json) through FreeCAD geometry generation to Blender presentation-quality rendering. Architects and stakeholders can iterate on the spec, regenerate models, and produce renders without manual Blender setup. The separation of run.sh (FreeCAD) from run_blender.sh (Blender) means the FreeCAD pipeline still works without Blender installed.

02 / Background & Objective

Background & Objective

The freecad-blender repo generates a 4×25m five-storey tube-house from a JSON specification using FreeCAD Python scripts. The research brief (research/2026-04-30_adding-blender-freecad-workflow.md) established that FreeCAD should remain the single source of truth for parametric geometry, while Blender serves as a downstream tool for visualization, material application, and high-quality rendering.

The objective was to build a reproducible, automated pipeline from FreeCAD's parametric model through to a renderable Blender scene, ensuring the FreeCAD-only path still works when Blender is unavailable.

04 / Assumptions & Constraints

Assumptions & Constraints

Assumptions

  • ASM-001: Blender 3.6+ is installed locally or available via blender on PATH for headless rendering.
  • ASM-002: OBJ preserves object group naming from FreeCAD labels so Blender can assign materials by convention.
  • ASM-003: The FreeCAD generation pipeline stays untouched — only export adapters and Blender scripts are added.
  • ASM-004: BlenderBIM and IfcOpenShell are future enhancements, not part of this initial integration.

Constraints

  • CON-001: Blender headless mode has no GPU for EEVEE; Cycles CPU is the default renderer.
  • CON-002: The FreeCAD Mesh module must be available for OBJ export beyond the existing STL path.
  • CON-003: OBJ group naming depends on consistent FreeCAD object labels; future label changes require material mapping updates.
05 / Methodology

Methodology

Each phase was implemented incrementally with pure-Python test coverage before Blender-dependent code. The approach followed the research brief's recommendation to retain FreeCAD as the source of truth and pipe outputs to Blender via OBJ, with JSON configuration driving material assignment rather than hardcoding.

Convention-based material inference (matching keywords in Blender object names to material categories) provides a zero-configuration default, while the JSON sidecar enables architects to customize materials without editing Python code. The pipeline is split into two independent runners — run.sh for FreeCAD and run_blender.sh for Blender — so each tool can be installed and run independently.

06 / Phase-by-Phase Analysis

Phase-by-Phase Analysis

PHASE-01: OBJ Export Adapter

Created src/blender_export_utils.py with per-floor and combined OBJ export functions. Objects are exported with named groups (g F0_Wall_ext_left_2D) and a 6-material MTL palette. A convention-based inference function (_infer_material_key) maps object labels to materials. Integrated into generate_floorplan.py via export_floor_obj() and export_combined_obj() calls. Added output/obj/ and output/blend/ directories.

Exit criteria met: All 19 pure-Python tests pass. OBJ export produces named groups. Existing FreeCAD workflow unchanged.

PHASE-02: Blender Scene Setup

Created src/blender_materials.py with 10 PBR materials (Principled BSDF), room/zone/OBJ-group material maps, and create_blender_material(). Created spec/blender_materials.json as a data-driven sidecar for material assignments, lighting (sun + fill area), camera (50mm lens targeting bounding center), and render settings (Cycles, 128 samples, 1920×1080). Created src/setup_blender_scene.py (import OBJ with STL fallback, assign materials, add lights/camera, save .blend) and src/render_blender.py (headless Cycles render). Created run_blender.sh batch runner.

Exit criteria met: All 35 tests pass. Scripts py_compile clean. JSON/Python material mappings are consistent.

PHASE-03: Batch Runner Integration

Verified the two-runner architecture: run.sh for FreeCAD (already triggers OBJ export via EXPORT_ARCHITECT_PACKAGE=1) and run_blender.sh for Blender. Updated run.sh to create output/obj/ and output/blend/ directories and display OBJ/MTL outputs in the success summary. Added a pointer to ./run_blender.sh in the run.sh output. Updated AGENTS.md with Blender pipeline entry points. Updated activeContext.md with complete state.

Exit criteria met: run.sh still works without Blender. run_blender.sh works independently. Failure at any stage produces a clear error message.

07 / Findings & Recommendation

Findings & Recommendation

Key Findings

  1. OBJ is the right interchange format for this use case. It preserves object group names that Blender can use for material assignment, requires no additional Python packages, and is natively importable by Blender. STL was available as a fallback but loses group structure.
  2. Data-driven material configuration via a JSON sidecar (spec/blender_materials.json) proved more flexible than hardcoding. Architects can tweak materials, lighting, and camera positions without touching Python.
  3. Convention-based fallback (keyword matching on object labels) provides zero-configuration material assignment, while the JSON sidecar overrides it when present.
  4. The two-runner architecture (separate run.sh and run_blender.sh) keeps the FreeCAD-only path clean and avoids confusing dependency errors when Blender is absent.

Recommendation

The pipeline is ready for runtime validation: install FreeCAD and Blender, run ./run.sh then ./run_blender.sh, and visually inspect the rendered output. Material and camera tuning will be the natural next iteration once renders are available.

IFC/BlenderBIM integration remains a future enhancement for when BIM property round-tripping or structural analysis becomes a requirement.

09 / Risks / Open Questions

Risks / Open Questions

Risks

  • RISK-001: Blender API changes between versions 3.6 and 4.x may break the scene setup or render script. Mitigated by targeting stable APIs and adding version checks.
  • RISK-002: OBJ group naming depends on FreeCAD label consistency. If labels change, material inference may misclassify. Mitigated by the JSON sidecar override.
  • RISK-003: Cycles CPU rendering at 128 samples × 1920×1080 may take 5-10 minutes on typical hardware. Mitigated by configurable sample count via BLENDER_RENDER_SAMPLES env var and JSON config.

Open Questions

  • What camera perspective best serves the architectural review — isometric, eye-level, or perspective?
  • Should the render pipeline integrate into CI, or remain a local developer tool?
  • When should IFC/BlenderBIM be considered for BIM property round-tripping?
